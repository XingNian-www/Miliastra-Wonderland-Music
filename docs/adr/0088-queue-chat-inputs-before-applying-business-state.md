# 按观察顺序排队聊天输入并独立选择效果通道

同一观察帧中被接受的普通聊天命令和稳定海龟汤提问，先按画面中的消息顺序合并，再作为正式输入任务进入 `FormalScheduler`。二级大厅按已发布消息的显示顺序提交同一类任务。扫描线程只完成观察、稳定判断、路由和入队，不直接推进娱乐业务状态；因此先出现的控制命令必须先于后出现的玩法输入生效。独占决策等待也不暂停这条普通输入路径：决策读者只消费保留的确认/投票语法，其他命令仍按一级屏幕锁或二级气泡基线去重后入队。命令可用性开关需要影响后续扫描，闲置退出只配置计时状态，监听模式使用专用切换事务，这三类运维控制继续保留已有专用路径。

输入任务进入正式 FIFO，不代表它产生的所有效果都属于 Formal 通道。普通短大厅回复进入 Deferred 通道；好友验证、私聊发牌、不可拆分批次和其他完整 UI 事务使用 Formal 通道。一个牌局效果链从首个效果到后续恢复必须保持同一通道，组合层按命令可能产生的完整效果选择端口。Formal 任务由 `FormalTaskClient` 以类型化提交边界入队，执行线程取出后才调用 `PendingTask`，扫描线程不直接持有业务服务。这样既保持输入状态转换的因果顺序，也保留 ADR 0061 规定的正式、延迟和诊断优先级。

排队输入携带观察帧的单调捕获时间。期限、回合和空闲活动以该时间判断，不以出队执行时间代替；控制台输入在接收时记录时间。海龟汤提问任务标签固定为“海龟汤提问”，不得把问题正文暴露到调度投影或任务历史。
