# 播放器控制器流程

本文说明播放器控制器的目标边界。播放器后端状态被视为不稳定观测，业务流程不能再只靠一次 `status()` 判断是否播放成功、是否应自动出队或是否该恢复播放。

## 核心结论

播放器控制器负责把播放意图、播放观测、活动播放请求、暂停原因和队列推进决策收敛到一个状态机里。`main.rs` 只负责命令入口、聊天确认、AI 判断、音乐播放队列数据操作和待执行任务编排。

FeelUOwn 只是当前播放器后端。后续如果接入其他播放器，应新增后端适配器实现同一组底层能力，而不是把新播放器逻辑写进点歌流程。

## 边界

| 模块 | 职责 |
| --- | --- |
| `player_controller` | 管理确认播放状态、活动播放请求、暂停原因、播放确认和队列推进决策。 |
| `MusicPlayerBackend` | 播放器后端 trait，提供状态查询、播放 URI、暂停、恢复、上一首、下一首、音量和候选搜索。 |
| `FeelUOwnClient` | FeelUOwn 的后端适配器，实现 `MusicPlayerBackend`。 |
| `main.rs` | 命令入口、候选确认、AI 同曲判断、聊天反馈、音乐播放队列持久化和待执行任务入队。 |
| `queue.rs` | 只保存音乐播放队列，不判断播放器状态。 |

## 状态模型

播放器控制器使用两层状态。

第一层是播放观测。它来自播放器后端，包含原始状态、URI、歌名、歌手、进度、时长和捕获时间。播放观测可能可靠、不完整、过期或与当前播放请求不匹配。

第二层是确认播放状态。它是业务状态，不会被单次异常观测直接覆盖：

| 状态 | 含义 |
| --- | --- |
| `Idle` | 当前没有控制器确认中的播放请求。 |
| `Starting` | 已发出播放意图，正在等待后端状态稳定。 |
| `RequestedSongPlaying` | 当前播放器已确认播放本次点歌请求。 |
| `PausedByUser` | 用户主动暂停，不允许自动恢复。 |
| `PausedWaitingForQueue` | 控制器为了等待队列或待执行点歌接管而暂停，空闲后可以自动恢复。 |
| `ExternalPlayback` | 播放器正在播放非本项目点歌请求；只有同一首连续正常播放达到外部保护延迟后才阻止新点歌接管。 |
| `Unknown` | 后端状态不可用或连续观测无法判断，不主动出队。 |

## 播放确认

主业务流程先把普通点歌、AI 点歌或远程点歌解析成最终候选。主路径应带有稳定 URI，然后交给播放器控制器播放。

控制器发出播放 URI 后进入 `Starting`，并在重试窗口里持续读取播放观测：

1. 看到旧歌、空 URI 或 `0:00/0:00` 时继续等待。
2. URI 与目标 URI 一致时确认成功。
3. URI 缺失或不一致时不能确认歌曲身份；标题和歌手只作为展示信息。
4. 本地判断仍不确定时，控制器返回结构化的不匹配结果，由主业务流程决定是否询问聊天或调用 AI。
5. 确认成功后写入活动播放请求，并记录同歌播放历史。

播放命令发送成功不等于播放确认成功。只有播放确认成功才算本项目实际播放了这首歌。

播放状态更新必须按事务语义处理：播放后端调用失败、播放确认超时或确认无音源时，控制器保留调用前的确认状态和活动播放请求；只有后续确认播放成功后，才提交新的确认状态。这样一次失败的外部 URI 播放或点歌 URI 下发，不会把仍在播放的 BOT 点歌误清成 `Idle` 或卡在 `Starting`。

## 队列推进

播放监控线程只把上下文交给播放器控制器，包括音乐播放队列是否为空、是否有待执行播放任务、命令是否正在执行、点歌命令是否正在执行。普通控制台发言、启动游戏、进入千星和管理投票这类非播放任务不会让控制器因为“临近结束”而暂停播放器。

控制器返回队列推进决策：

| 决策 | 调用方动作 |
| --- | --- |
| `None` | 不做事。 |
| `PlaybackStateChanged` | 刷新监控里的播放器控制器状态。 |
| `PauseForQueue` | 控制器暂停播放器并记录 `PausedWaitingForQueue`。 |
| `ResumeIfIdle` | 只有暂停原因是 `PausedWaitingForQueue` 时恢复播放。 |
| `AdvanceQueue(reason)` | `main.rs` 把 `PendingTask::AdvanceQueue` 放入待执行任务队列。 |

用户主动暂停和系统临近结束暂停必须分开。用户暂停不会自动恢复；系统暂停只在等待队列接管时使用。

## HTTP 与监控

播放相关 HTTP 入口不能直接调用播放器后端修改状态。远程播放控制走待执行任务队列，`/player/play-uri` 则表达“把指定 URI 放入音乐播放队列”的业务意图。该入口不会直接改变播放器状态；真正播放时仍由音乐播放队列和播放器控制器确认。

监控快照可以展示控制器状态，例如确认播放状态、暂停原因、活动播放请求和最近一次观测可靠性。普通日志只记录状态转移和最终决策，逐次观测放在调试日志中。
