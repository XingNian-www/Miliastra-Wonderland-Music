# 二级聊天监听

二级聊天监听是运行期模式，不写入 `config.yaml`。它复用现有游戏窗口聚焦、模板匹配、OCR 和待执行任务队列，但不复用一级聊天的命令屏幕锁。

## 模式入口

好友私聊可使用以下高级命令：

| 命令 | 作用 |
| --- | --- |
| `@监听模式 一级` | 切换回原有一级聊天扫描。 |
| `@监听模式 二级` | 进入二级聊天监听。 |
| `@监听模式 状态` | 只记录当前模式和是否有切换任务等待执行。 |

当前大厅的 `blue` 消息不能切换模式。Web 面板通过 `POST /chat-listener/mode?mode=primary` 或 `secondary` 提交同一类待执行任务；`/monitor.chatListener` 提供当前模式和等待目标。TUI 只显示状态，不提供操作入口。

## 进入与回退

切换到二级监听由 `PendingTask::SetChatListenerMode` 串行执行，流程如下：

1. 激活并聚焦游戏窗口。
2. 如果当前不在二级界面，按一次 `Enter`。
3. 点击左侧“当前大厅”模板。
4. OCR 顶部标题，确认是“当前大厅”。
5. 成功后进入首次未读清场；失败则第二次尝试。

第二次尝试前先用 `Esc` 返回一级界面。两次都不能确认“当前大厅”时，模式恢复为一级监听。

普通命令不再统一以一级界面作为起点。执行器先激活游戏，然后由每个动作阶段声明自己需要的界面；业务真正结束后再恢复监听模式的驻留界面。二级监听的驻留目标是“二级当前大厅”，一级监听的驻留目标是一级界面。

## 命令动作状态机

命令来源不决定执行界面。命令排队期间游戏画面可能已经变化，因此执行时必须重新观察实际界面，并按当前动作的目标做最短转换。

| 动作阶段 | 目标界面 | 转换规则 |
| --- | --- | --- |
| 查询播放器、修改队列 | 无固定游戏页面 | 不预先按 `Esc`；需要回复时再进入监听驻留界面。 |
| 二级大厅回复 | 二级当前大厅 | 已在当前大厅则直接发送；在其他二级会话时点击“当前大厅”；在一级时按 `Enter` 后点击“当前大厅”。 |
| 大厅信息 OCR、麦克风、管理面板按键 | 一级界面 | 该动作开始前才逐级 `Esc` 到一级，不影响同一命令的其他阶段。 |
| 好友回复、邀请入口 | 二级好友会话 | 已在目标好友会话时直接复用；否则只打开二级聊天并点击好友，不先绕回一级。 |
| 命令结束 | 当前监听模式的驻留界面 | 二级模式确认“当前大厅”，一级模式确认一级界面，确认成功后才算完成。 |

扫描线程看到命令任务临时离开二级界面时不会降级监听模式。任务失败也先尝试恢复驻留目标；只有二级当前大厅恢复失败时才回退一级监听并发送异常信息。目标窗口不可用时不再发送任何键鼠输入，当前业务直接失败。

点歌、邀请和仅接受大厅消息的自定义确认，在二级模式下直接读取“当前大厅”新增的深色气泡。确认读取器先记录现有气泡序列，再以尾部和头部的最长重叠确定新增气泡；轮询间隔限制在 `100-500ms`，新增气泡稳定后才 OCR。需要接收粉色好友私聊的确认仍临时进入一级扫描，以覆盖多个会话。

管理投票是跨任务状态机：投票公告发出后建立“临时一级阶段”，后台线程在一级界面同时收集多个好友表决。投票结果任务携带这个阶段的持有对象；多个投票并行时采用计数，最后一个投票结束后才恢复二级当前大厅。Web/TUI 会显示“二级监听（临时一级阶段）”。

## 二级会话分类

顶部标题只用于判断来源，不用于严格校验好友昵称：

| 顶部标题 | 命令来源 | 行为 |
| --- | --- | --- |
| `当前大厅` | `blue` | 处理当前大厅可见气泡序列中新出现的气泡。 |
| `公开频道` | 无 | 加入“恢复当前大厅”任务。 |
| `陌生人消息` | 无 | 不执行命令。 |
| 其他任意标题 | `pink` | 视为好友私聊，即使昵称 OCR 有误。 |

好友未读路径以红点消失作为成功确认，因此顶部昵称无法准确识别时，仍可以按 `pink` 解析命令。无法自动回到特定好友私聊发送反馈时，沿用大厅异常反馈，不维护长期私聊会话句柄。

## 图像探针与 OCR

所有识别区域都使用项目截图和输入动作共用的 `1920x1080` 游戏客户区坐标。用于讨论界面的 Windows 截图如果包含窗口标题栏，标题栏不属于游戏画布，不能把截图中的纵坐标直接写入识别逻辑。

扫描器使用两个低成本图像探针：

- 左侧好友列表只扫描头像右上角的窄红色区域，按红色占优和连通区域大小识别未读红点；不会扫描全屏，也不会自动滚动好友列表。
- 消息区只识别深色他人气泡。当前大厅按从上到下的完整气泡序列定位消息，OCR 只覆盖气泡本体，不把昵称标题纳入识别；自身浅色气泡不会触发命令 OCR。

发现气泡变化后，程序会每隔约 `100-200ms` 检查一次最新深色气泡摘要；连续稳定后 OCR，最长等待约 `500ms`，超时则使用最后一帧 OCR。这样可以覆盖文字逐步出现和聊天动画，同时不会长时间卡住监听循环。

首次进入二级监听时，当前可见的好友红点会逐个打开并清除，但不 OCR、不解析、不入队。完成后当前大厅的最新深色气泡只建立基线。屏幕外好友不主动滚动；只有他们之后发言并自然进入可见区，才会被正常处理。

正常运行时，红点优先级高于普通大厅扫描，但每次只处理一位好友：

1. 点击红点对应好友行，最多两次。
2. 红点消失后，等待气泡稳定。
3. 只 OCR 最下方最新的他人气泡。
4. 把可解析内容作为 `pink` 命令加入待执行任务队列。
5. 恢复当前大厅，并至少扫描当前大厅一轮后才处理下一位好友。

当前大厅保留完整可见气泡序列的相对图像摘要。每轮把旧序列尾部与新序列头部做最长重叠匹配；重叠后的全部新气泡按从上到下顺序 OCR 和入队，因此不会漏掉同一屏连续出现的多条大厅命令。找不到可靠重叠时只重建基线，不执行当前可见内容，避免滚屏或动画将历史消息重复执行。好友未读路径仍只 OCR 最下方最新气泡。

## 并发边界

扫描线程只做探针、OCR 和任务入队。点击好友、清除红点、恢复当前大厅以及切换监听模式全部是 `PendingTask`，由现有命令执行线程串行执行。这样二级监听不会与邀请、拉黑、点歌或控制台发言并发抢占键鼠。

二级监听的命令不经过一级 `CommandLockState`：红点消失和最新气泡摘要分别承担“旧未读已清场”和“当前画面是否变化”的判断。待执行任务队列仍会拒绝同语义的未执行任务。
