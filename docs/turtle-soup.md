# 海龟汤

海龟汤是默认关闭的大厅娱乐模块。它与成语接龙、斗地主共用全局娱乐互斥，同一时间只能运行其中一种玩法；控制命令和稳定提问按观察顺序进入正式 `PendingTask` 队列，AI 裁决在独立 Worker 中进行，游戏内回复只在正式任务完全空闲时发送。

## 游戏命令

当前大厅蓝字支持：

```text
#海龟汤
#状态
#结束
#他认识死者吗？
##1男人是灯塔管理员
##2他关灯后导致船只失事
##提交
```

- 空闲时发送 `#海龟汤` 会随机选择一题并开局；进行中再次发送会完整重发汤面。
- `#状态` 回复持续时间、参与人数和有效提问数。
- `#结束` 主动结束当前对局。
- 只有正文以 `#` 开头的当前大厅蓝字才是提问；消息中的普通 `#` 不会触发。
- `##编号内容` 按昵称和编号暂存长答案，不调用 AI；成功后只回复 `暂存[昵称]:##编号`，不回显正文。
- 编号必须从 `1` 开始且在提交时连续；可以乱序发送，同一编号再次发送会覆盖旧内容。
- `##提交` 校验编号后按顺序用换行合并，并作为一个当前问题只调用一次 AI；缺段时只反馈第一个缺少的编号并保留草稿。
- `##取消` 清空当前昵称的草稿。普通 `#问题` 始终立即独立裁决，不会加入批量草稿。
- 第一位发送有效提问的昵称自动成为参与者，不需要加入命令。
- Web 控制台也可以结束当前对局。

一级大厅先分别确认昵称和正文连续 OCR 一致，再按“玩家昵称 + 输入类型 + 批量编号 + 问题正文”锁住当前可见提问。两项默认继承 `stability.default_count`；`turtle_soup.nickname_stable_count` 和 `turtle_soup.content_stable_count` 只有大于 1 时才分别覆盖，0 或 1 表示继承，两级都无有效值时使用内置默认 2 次。问题正文同时作为昵称 OCR 的辅助锚点：只有输入类型和问题相同，并且规范化昵称最多相差一个字符时，才视为同一条消息。两个明显不同的昵称即使问了相同问题，仍会分别处理。昵称或正文实际变化只重置对应稳定计数，两项都达标后才会形成正式提问任务，不在扫描线程暂存或提交 AI。比较会统一空白、大小写、全半角和常见标点，因此 OCR 把同一句末尾的 `？` 识别成 `?` 时不会重置稳定计数或再次提交；正文、输入类型或批量编号实际变化时仍视为新输入。已锁定问题也必须连续不可见达到同样的最大稳定次数才会解锁，单轮漏掉 `#` 或聊天标记不会让恢复后的旧问题再次提交。启动或进入新大厅后的首批可见提问只建立基线，不会在达到稳定次数后补执行。送给 AI 的问题保留最终确认的 OCR 文本，标准化只用于稳定判断和去重。

同一帧中，稳定提问与控制命令按画面从上到下合并入队；位于控制命令之后的问题不能先执行。提问刷新空闲期限时使用该观察帧的捕获时间，不能用排队后的执行时间延长会话。调度器只显示固定任务标签，不显示问题正文。

二级当前大厅在海龟汤提问阶段会对整批新增深色气泡重复截图和 OCR。正文尚未达到稳定次数时不会推进气泡基线；稳定正文以 `#` 开头时，还必须让对应昵称达到独立稳定次数。这样第一次识别不稳定的气泡会在下一轮继续复核，不会被当成已读后丢失。稳定正文不是海龟汤输入时，仍按普通二级命令处理。

大厅或 Web 主动结束、空闲超时、达到单局最长时间、玩家完整还原真相时，机器人都会在当前大厅分段发送结算摘要和完整汤底。游戏窗口关闭、进入千星或邀请流程进入新大厅时，会话直接中止、取消旧 AI 结果且不向新大厅泄露旧汤底。

## 题库与永久使用记录

正式题库路径由 `turtle_soup.question_bank_path` 指定，默认是被 Git 忽略的 `turtle_soup.yaml`。格式参考根目录的 `turtle_soup.example.yaml`：

```yaml
题目:
  - id: example-001
    标题: 仅供 Web 管理查看的标题
    汤面: 游戏内原样分段发送的汤面
    汤底: 结算时原样分段发送的汤底
    裁决备注: 供 AI 判断核心事实的补充口径
    启用: true
```

每次开局都会重新读取并严格校验整个题库。任一题缺少字段、`id` 重复、关键文本为空或 YAML 格式错误，整次开局都会失败。

题目选中后，程序先把 `id` 原子写入 `turtle_soup.used_state_path`，再开始发送汤面。已经使用的题目永久排除，即使汤面发送失败也不会恢复。程序、Web 和 API 都不提供重置操作；需要重置时只能停机后手动修改使用记录文件。

外部工具可通过 `POST /turtle-soup/questions` 提交已经整理好的单题 JSON：

```json
{
  "title": "标题",
  "surface": "汤面",
  "bottom": "汤底",
  "adjudicationNotes": "裁决备注",
  "enabled": true
}
```

主程序按收到写入锁的顺序为题目分配 `soup-0001` 形式的 ID，校验完整题库后原子替换本地 YAML。请求正文上限 64 KiB，沿用 `X-Miliastra-Token` 鉴权且不会写入 HTTP 历史。主程序不负责 AI 改写；配套云崽插件先完成内容整理，再提交最终结构。

## AI 裁决

海龟汤使用独立的 `turtle_soup.ai` Provider，不复用点歌 AI 或候选歌曲审核，也不启用联网搜索。当前配置模板使用 OpenAI 官方 `https://api.openai.com/v1/chat/completions` 和 `gpt-5.6`。请求只包含 OpenAI Chat Completions 标准字段；确有兼容需要时可通过 `turtle_soup.ai.extra_body` 显式补充第三方字段，与官方字段重名时官方值优先。`turtle_soup.ai.http_proxy` 可单独指定 HTTP(S) 代理，留空时沿用环境代理设置。其他兼容标准 Chat Completions 的 Provider 仍可通过显式修改 endpoint、model 和 `extra_body` 使用。

`max_tokens` 是当前配置必填字段，模板值为 `1024`，直接控制单次裁决最大输出 Token。请求固定使用 `temperature: 0.0`、`response_format: json_object`、`stream: false` 和 `store: false`，不设置 `top_p`。固定核心提示词明确要求合法 JSON 并提供输出示例；空内容、非法 JSON、非法裁决以及 `finish_reason=length` 都按请求失败进入重试。DeepSeek 的历史调研见 [海龟汤 AI 接入 DeepSeek 官方 API 的推荐配置](research/deepseek-turtle-soup-api.md)。

旧版私有关闭思考参数、`max_tokens=256` 和基础决策树曾经过 553 次 DeepSeek 官方 API 请求验证。当前标准化请求不再发送私有思考参数，因此该历史结果不能直接证明当前模型行为；详细数据和适用范围见 [DeepSeek 海龟汤裁决稳定性评测](research/deepseek-turtle-soup-evaluation.md)。

固定核心提示词负责约束 JSON 和裁决含义，`turtle_soup.custom_prompt` 只追加房间自己的裁决口径。

裁决结果只有：是、否、无关、部分正确、完全正确、拒绝、审查失败。普通裁决统一采用 `[昵称]的{问题摘要}回复：{结果}`，不发送模型的自由解释。问题超过 8 个字符时，摘要取去除空白后的前 4 个字符和后 4 个字符，中间使用 `....`，例如 `[星念]的男人是灯....理员吗？回复：否`；不超过 8 个字符时保留完整问题。游戏昵称最多 7 个字符，固定摘要格式下整条回复不会超过 80 显示宽度，也就是 40 个全角字符；代码仍保留长度保护作为异常输入兜底。“完全正确”直接进入获胜结算，不额外发送普通裁决短句。

裁决会把相关命题分成“准确成立”“相容近似”和“错误”。相容近似必须与汤底相容，只是概括过宽、过窄、缺少细节或措辞不专业；与汤底矛盾的具体身份、对象、地点、动作或原因属于错误。只要相容近似存在，或正确关系与错误关系同时存在，就判为部分正确；相关命题全部错误且没有相容近似时判为否。发言中的命题都准确成立、只是尚未还原完整汤底时判为是。

“完全正确”必须再进行一次不携带首次结论的独立复核。两次都返回完全正确才进入获胜结算；复核不一致按部分正确继续，复核技术失败按审查失败继续，不会结束对局。

默认并发为 4，每个昵称最多保留一条处理中请求，等待队列最多 32 条。单次请求超时 20 秒，失败重试 2 次。每次 AI 请求只包含汤面、汤底、裁决备注和本次问题，不包含昵称、其他玩家问题或历史裁决，因此并发完成顺序不会改变模型上下文。依赖上一问且无法从题目资料独立理解的问题按“该问题无法裁决”回复。

批量长答案每个昵称独立暂存，默认最多 `32` 段，由 `turtle_soup.batch_max_parts` 调整。片段不占 AI 等待队列，也不增加有效提问数；只有 `##提交` 校验通过并成功入队后才记为一次提问。AI 队列已满或该昵称仍有问题正在裁决时，草稿会保留以便稍后重新提交。换题、结束或中止对局时，所有未提交草稿都会清除。

每次开局都会先校验独立 AI Provider 的 HTTP(S) 地址、密钥和模型。校验未通过时不会读取题库、占用娱乐互斥或把题目标记为已使用。

日志只记录请求 ID、昵称、最终裁决、耗时、重试次数和错误摘要。日志不会记录汤底、原始 Provider 响应或模型内部理由。

## 分段与调度

游戏聊天上限按 80 显示宽度计算，也就是最多 40 个全角字符。汤面和汤底不会走普通截断逻辑，而是完整拆成：

```text
汤面1/3：...
汤面2/3：...
汤底1/4：...
```

编号和前缀也计入宽度，不限制总段数。发送器会把同一批剩余分段交给聊天批量发送接口：一级大厅只打开一次聊天，二级当前大厅直接复用已打开的聊天界面。延迟发送线程只在正式任务空闲时开始批次；批次开始后独占游戏输入直到全部完成或某条发送失败，新到达的正式任务等待本批结束。

开局汤面、汤面重发与结算批次属于受保护回复，不会被普通 32 条延迟回复容量挤掉。开局与结算属于更高优先级的关键批次，会从延迟回复队列头部进入；结算开始时还可以淘汰尚未开始发送的旧汤面重发，确保最终汤底优先。首条未发送消息最多尝试 3 次；批量调用在中途失败时，只给这条未发送消息累计失败次数，之前成功的消息永久保留为已完成。开局汤面失败会终止本局但保留已使用记录；汤面重发失败只停止本次重发；普通裁决失败只丢弃该回复；汤底或结算摘要失败会停止剩余结算、释放娱乐互斥，并在 Web 快照中标记发送不完整。

## Web 控制

主 Web 面板显示状态、汤面、题目标题和 ID、开局者、时长、参与人数、有效提问数、AI 待处理数、剩余题目数和最近 30 条裁决。最近裁决只用于监控，不会再次发送给 AI；Web 接口不会返回进行中的汤底。

| 接口 | 方法 | 行为 |
| --- | --- | --- |
| `/turtle-soup` | GET | 读取当前海龟汤快照。 |
| `/turtle-soup/start` | POST | 从未使用题目中随机开局。 |
| `/turtle-soup/start?id=...` | POST | 按未使用且已启用的题目 ID 开局。 |
| `/turtle-soup/end` | POST | 主动结束并在当前大厅公布结算和汤底。 |
| `/turtle-soup/questions` | POST | 串行追加一条已经整理好的题目并返回 ID 和位置。 |

没有在线编辑、题库替换或使用记录重置接口。
