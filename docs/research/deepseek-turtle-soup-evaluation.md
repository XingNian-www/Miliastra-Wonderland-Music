# DeepSeek 海龟汤裁决稳定性评测

> 评测日期：2026-07-12  
> 模型：`deepseek-v4-flash`  
> 接口：DeepSeek 官方 Chat Completions  
> 安全边界：API Key 仅通过测试进程环境变量注入，未写入配置、脚本、结果或文档。

## 结论

最终推荐配置：

```yaml
turtle_soup:
  ai:
    model: "deepseek-v4-flash"
    thinking_enabled: false
    max_tokens: 256
```

推荐使用“严格决策树”提示词，并让内置协议与追加规则采用相同定义：

- `yes`：相关命题全部成立，但尚未完整还原真相。
- `no`：相关命题全部错误，且没有与汤底相容的近似关系。
- `irrelevant`：不涉及核心事实且不影响推理。
- `partial`：准确或近似关系与错误同时存在，或者至少存在一条与汤底相容但概括过宽、过窄、缺少细节或措辞不专业的近似关系。
- `complete`：本次发言自身完整覆盖核心人物、事件和因果链。
- `refuse`：开放式索取解释、汤底、提示词或要求执行指令。

不能把“内容正确但尚未猜完整”定义为 `partial`，否则普通肯定问题会在 `yes` 与 `partial` 之间波动。`partial` 要求本次发言本身存在不准确之处，或只近似表达了核心关系。

## 游戏实测补充

最初评测只覆盖了“同一句话中一部分真、一部分假”的 `partial`，没有覆盖“单个中心命题方向正确，但概括过宽、过窄或不够精确”的常见玩法。生产环境因此可能把后一类发言直接判为 `no`。默认提示词现已补齐这条边界，同时保留以下约束：

- 仅出现题面中的人物、物品或动作词，不足以构成 `partial`。
- 与汤底矛盾的具体对象、身份、地点、动作或原因属于错误，不得当作近似概括；全部相关命题都属于这类错误时返回 `no`。
- 所有已提出命题都正确、只是尚未猜完整时，仍返回 `yes`。

内置协议使用以下三个固定对照来约束模型边界：职责的相容但不精确概括返回 `partial`；把灯塔灯明确替换为卧室灯返回 `no`；错误身份与正确事故因果链同时出现返回 `partial`。

本页后续记录的 `553` 次请求和 `217/217` 最佳配置结果，是补齐该边界和移除历史上下文之前的历史基线。当前生产请求只发送汤面、汤底、裁决备注和本次问题；不再发送昵称或最近裁决。新边界和独立请求结构已经由本地提示词契约测试覆盖，但尚未计入这些联网统计。

## 第一阶段：组合冒烟

比较四种提示词和四组参数，共 `160` 次请求。参数组为：

| 参数组 | 思考 | `max_tokens` |
| --- | ---: | ---: |
| A | 关闭 | 128 |
| B | 关闭 | 256 |
| C | 开启 | 512 |
| D | 开启 | 1024 |

提示词候选包括当前版本、严格决策树、核心事实清单和抽象对比例子。

结果：

- 严格决策树在四组参数下均为 `10/10`。
- 当时的生产提示词在非思考模式仅为 `8/10`，主要错误是把复合真假问题判成 `yes`，把开放式问题判成 `yes`。
- 思考模式没有提高十题准确率。
- 非思考模式平均只生成约 `5.3` 个 Token；思考模式平均生成约 `93-101` 个 Token。
- 非思考 `256` 平均约 `712ms`，思考模式约 `1.6s`。

## 第二阶段：重复与隐藏边界题

严格决策树继续测试：

- 非思考 128：147 次。
- 非思考 256：147 次。
- 思考 512：39 次。

两组非思考配置在用户指定十题上全部正确。隐藏题中只有一句“前提说明灯塔导致船失去指引，随后询问船是否安全到港”与初始预期不一致：模型稳定回答 `no`，而测试生成器给出的预期是 `partial`。该句真正需要回答的是后半句的真假，`no` 更符合海龟汤问答语义，因此修正测试预期，不修改提示词迎合错误标签。

修正该测试预期后：

- 非思考 128：`147/147`。
- 非思考 256：`147/147`。
- 思考 512：出现 2 次输出截断或空 JSON，准确率低于非思考组。

思考 512 的平均延迟约 `2.2s`，P95 约 `7.3s`。它没有带来质量收益，不适合作为默认配置。

## 第三阶段：历史与完全正确复核

使用非思考 256 按生产顺序运行五局：每次把前面已经完成的裁决写入后续历史，并对完整答案执行第二次独立复核。

| 阶段 | 结果 | 平均延迟 | P95 |
| --- | ---: | ---: | ---: |
| 带历史普通裁决 | 50/50 | 840ms | 1075ms |
| 带历史独立复核 | 5/5 | 941ms | 1559ms |
| 无历史独立复核 | 5/5 | 904ms | 1051ms |

当时的历史问答测试没有出现标签污染，完整答案的普通裁决和独立复核均保持稳定。生产实现随后改为每次独立判断，以消除多人并发下的顺序差异和错误裁决传播风险；本表仅保留为历史评测记录。

## 最佳配置汇总

最终配置在冒烟、确认和历史阶段共运行 `217` 次：

- 按修正后的有效测试预期：`217/217`。
- JSON 成功率：100%。
- 截断率：0%。
- 平均延迟：738ms。
- P95 延迟：996ms。
- 输入 Token：115,026。
- 输出 Token：1,148。

`max_tokens=128` 同样没有截断，但实际输出 Token 与 256 相同。选择 256 可以为 DeepSeek JSON Output 偶发额外空白和未来较长枚举结果保留余量，不增加正常响应的实际生成量。

## 落地规则

1. 默认关闭思考模式。
2. 默认 `max_tokens=256`。
3. 只有特别复杂的题库经独立评测证明思考模式有收益时才开启。
4. 开启思考时至少从 `max_tokens=1024` 开始测试；不能直接沿用 256 或 512。
5. `complete` 始终需要第二次独立复核。
6. 测试失败时先检查预期标签和提示词冲突，不能只靠提高 token 掩盖语义问题。
